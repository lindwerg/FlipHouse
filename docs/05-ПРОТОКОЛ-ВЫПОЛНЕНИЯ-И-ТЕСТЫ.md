# FlipHouse — Протокол выполнения и тесты

> Операционный регламент сборки. Как прогонять роадмап (`roadmap/P0..P7`) через ultracode так,
> чтобы founder'ская цель №1 — **ZERO bugs** — выполнялась механически, а не на удачу.
> Версия 1.0 · 2026-06-15 · Деплой-цель: Railway (Pro)
>
> **Главный закон:** шаг не существует как «сделанный», пока его тесты не зелёные и coverage-гейт держится.
> Никакого «потом допишу тест». Никакого «вроде работает». Зелёный CI — единственное доказательство.

---

## 0. Как это читать

Этот документ — **исполнительный контракт** между founder'ом и исполнителем (ultracode-агентом).
Он не описывает *что* строить (это делают `roadmap/P0..P7` и `docs/00..04`), он описывает *как именно гонять*:

- **§1** — инструменты тестирования, команды, пороги покрытия.
- **§2** — жёсткий чек-лист одного шага (RED→GREEN→REFACTOR→COMMIT).
- **§3** — протокол чекпоинта (когда исполнитель ОБЯЗАН остановиться и ждать founder'а).
- **§4** — готовый ШАБЛОН ПРОМПТА для вставки в ultracode (запуск следующего шага).
- **§5** — гейт релиза (что должно пройти перед деплоем на прод).
- **§6** — правило вмешательства (как founder правит фазу между шагами и как исполнитель подхватывает).

Единый источник истины о прогрессе — **`STATE.md` в корне репо**. Он заводится в Шаге 0.0 (см. `roadmap/P0`).
Всё, что ниже, читает и пишет `STATE.md`.

## 1. Тест-пирамида, инструменты, команды, пороги

### 1.1. Пирамида (снизу вверх)

```
                ┌─────────────────────────────┐
                │   E2E / smoke всего пути     │  Playwright (web) + smoke-скрипт пайплайна
                │   (мало, медленно, дорого)   │  → только критические user-flow
                ├─────────────────────────────┤
                │   Integration / contract     │  Vitest (API-роуты, BullMQ), pytest (render),
                │   (API, очередь, рендер)     │  golden-file asserts (детерминированный вывод)
                ├─────────────────────────────┤
                │   Unit (много, быстро)       │  Vitest (TS), pytest (Python) — чистые функции,
                │   (фундамент пирамиды)       │  утилиты, matching, хелперы
                └─────────────────────────────┘
```

Правило формы: **много быстрых юнитов, мало медленных e2e**. Если фича тестируется юнитом — она НЕ тестируется e2e.
E2E резервируем для сквозных путей, которые ломаются на стыках (auth→upload→clip→insert→publish).

### 1.2. Инструменты по слоям

| Слой / пакет | Харнесс | Что покрывает | Тип в пирамиде |
|---|---|---|---|
| `packages/shared` (TS) | **Vitest** + `@vitest/coverage-v8` | content-hash, matching, чистые утилиты | unit |
| `apps/web` (Next.js) | **Vitest** (unit) + **Playwright** (e2e/smoke) | UI-логика, API-роуты, auth/billing flow, лендинг | unit + e2e |
| `apps/worker-node` (BullMQ) | **Vitest** | оркестрация очереди, idempotency, retry, контракты job'ов | unit + integration |
| `services/ai-worker-python` | **pytest** + `pytest-cov` | transcribe→clip→reframe→caption→banner; детерминизм рендера | unit + **golden-file** |
| Кросс-сервисный путь | **Playwright** + **smoke-скрипт** | сквозной user-flow на staging | e2e |

**Golden-file asserts (рендер-детерминизм):** Python-воркер на фиксированном входном фикстур-видео + фиксированном seed
обязан давать **бит-в-бит/структурно стабильный** выход. Тест сравнивает результат с зафиксированным golden-артефактом:
- для JSON-контрактов (тайминги слов, координаты реврейма, план вставки оффера) — точное сравнение с golden-JSON;
- для видео/кадров — сравнение по структурному хешу ключевых кадров / манифесту (а не пиксель-в-пиксель, чтобы не ловить шум кодека), порог расхождения = 0 на контрактных полях.
Любой дрейф golden → **RED**. Обновлять golden можно только осознанным коммитом `test(...): update golden` с объяснением в теле.

### 1.3. Команды запуска

Все команды — из корня репо `/Users/mishanikhinkirtill/Desktop/FlipHouse`. Агрегаторы заведены в корневом `package.json` (Шаг 0.1).

```bash
# --- быстрый цикл одного шага (то, что гоняешь в RED/GREEN) ---
pnpm vitest run <pattern>                       # точечный TS-юнит по имени/паттерну
pytest services/ai-worker-python -k "<expr>"    # точечный Python-тест
pnpm --filter @fliphouse/shared coverage        # coverage конкретного пакета (видеть exit-code)

# --- полный локальный прогон (перед коммитом шага) ---
pnpm lint                                        # ESLint (TS) + Ruff/Black --check (Python через скрипт)
pnpm typecheck                                   # tsc --noEmit по всем пакетам
pnpm test                                        # все Vitest-проекты + pytest (через -r агрегатор)
pnpm coverage                                    # агрегированное покрытие + ПОРОГИ (роняет билд)
pnpm test:e2e                                    # Playwright (web) — локально против dev-сервера

# --- гейт релиза (см. §5) ---
pnpm test:all                                    # lint + typecheck + test + coverage + e2e одной командой
pnpm smoke -- --base-url=<staging-url>           # сквозной smoke всего пути на staging
```

> Если какого-то агрегата ещё нет (ранние шаги P0) — он заводится тем же шагом, что и первый тест слоя.
> Не изобретать команды: имена скриптов фиксированы в `roadmap/P0` Шаг 0.1 (`lint`,`typecheck`,`test`,`test:e2e`,`coverage`) и расширяются (`smoke`,`test:all`) по мере фаз.

### 1.4. Пороги покрытия (coverage thresholds)

| Цель | Порог | Где настроено | Поведение при недоборе |
|---|---|---|---|
| `packages/shared` (TS) | **≥ 80%** statements / branches / functions / lines | `packages/shared/vitest.config.ts` → `test.coverage.thresholds` | `exit ≠ 0`, билд падает |
| `services/ai-worker-python` | **≥ 80%** | `pyproject.toml` → `[tool.coverage]` + `--cov-fail-under=80` | `pytest` exit ≠ 0 |
| `apps/web` (unit) | **≥ 80%** на бизнес-логике (роуты/хуки/утилиты; presentational-компоненты покрываются Playwright'ом) | `apps/web/vitest.config.ts` | `exit ≠ 0` |
| `apps/worker-node` | **≥ 80%** | `apps/worker-node/vitest.config.ts` | `exit ≠ 0` |

**Гейт обязан реально РОНЯТЬ билд, а не варнить.** Это доказывается отдельным «proof-of-failure» тестом
(см. `roadmap/P0` Шаг 0.4): временно добавить непокрытый экспорт → `coverage` даёт `exit ≠ 0` → убрать.
Порог можно поднять (80→85/90) на ЧЕКПОИНТЕ B, но никогда не опускать без явного решения founder'а в `STATE.md`.

### 1.5. Анти-флей правила (web-testing.md)

- E2E: **детерминированные ожидания** (`expect(locator).toBeVisible()`, `waitForResponse`), НЕ `waitForTimeout`.
- Визуальная регрессия (где есть UI-вес): Playwright-скриншоты на брейкпоинтах 320/768/1024/1440; обе темы, если обе существуют.
- Reduced-motion и контраст проверяются в e2e критических экранов.
- Тесты изолированы: своя БД-транзакция/мок, никакого общего состояния между тестами.

## 2. Цикл одного шага — жёсткий чек-лист RED→GREEN→REFACTOR→COMMIT

> Это **единица работы**. Исполнитель делает РОВНО ОДИН незакрытый шаг из текущего файла фазы за проход.
> Шаг закрывается только при прохождении ВСЕХ пунктов ниже. Нельзя начинать следующий шаг, не закрыв текущий.

### Чек-лист (нельзя пропустить ни один пункт)

```
[ ] 0. ВЗЯТЬ ШАГ
       └─ Прочитать в файле фазы блок текущего шага: Цель/DoD, «Тесты СНАЧАЛА», «Реализация», «✅ Готово когда», «Commit».
       └─ Если шаг помечен 🛑 ЧЕКПОИНТ — выполнить тело шага, затем уйти в §3 (остановка), НЕ начинать следующий.

[ ] 1. RED — написать падающий тест ПЕРВЫМ
       └─ Создать тест с ТОЧНЫМ именем и assertion'ом из блока «Тесты СНАЧАЛА».
       └─ Запустить точечно (pnpm vitest run <pattern> / pytest -k ...).
       └─ Убедиться, что он ПАДАЕТ по правильной причине (фича отсутствует), а не из-за опечатки/импорта.
       └─ ❌ Если тест зелёный сразу — он бесполезен. Переписать, чтобы реально проверял поведение.

[ ] 2. GREEN — минимальная реализация
       └─ Написать МИНИМУМ кода, чтобы тест(ы) шага позеленели. Никакой спекулятивной обобщённости (YAGNI).
       └─ Запустить точечно → GREEN.
       └─ Запустить весь слой (pnpm test или pytest пакета) → не сломать соседние тесты.

[ ] 3. REFACTOR — улучшить, не ломая зелёное
       └─ Иммутабельность, мелкие файлы (200–400 строк, max 800), ранние возвраты, именованные константы.
       └─ Убрать дубли (DRY), упростить (KISS). Удалить console.log/debug.
       └─ После КАЖДОЙ правки — перезапуск тестов слоя → остаётся GREEN.

[ ] 4. GATES — прогнать локальные гейты целиком
       └─ pnpm lint        → 0 ошибок
       └─ pnpm typecheck   → 0 ошибок
       └─ pnpm test        → 0 failed (все слои)
       └─ pnpm coverage    → пороги держатся (exit 0); если шаг добавил код — покрытие не упало ниже порога
       └─ (если шаг трогает UI/flow) pnpm test:e2e → зелёный
       └─ ❌ Любой красный гейт = шаг НЕ закрыт. Чинить реализацию, НЕ тест (если тест не доказанно неверен).

[ ] 5. ПРОВЕРКА «✅ Готово когда»
       └─ Пройтись по списку «✅ Готово когда» из блока шага буквально. Каждый пункт — отдельная галка.
       └─ Где требуется ручная проверка (например proof-of-failure) — выполнить и зафиксировать результат.

[ ] 6. STATE.md — обновить журнал
       └─ Дописать строку в «Журнал шагов»: phase.step | commit-type | дата | что сделано | тесты зелёные? (✅)
       └─ Обновить «Текущая фаза» и «Следующий шаг» (= следующий незакрытый шаг файла фазы).
       └─ Если всплыл риск — записать в «Открытые риски».

[ ] 7. COMMIT — один атомарный коммит
       └─ Сообщение РОВНО как в поле «Commit» блока шага: <type>: <message> (feat/fix/test/chore/ci/docs/perf).
       └─ В коммит входит: тест + реализация + обновлённый STATE.md. Ничего лишнего.
       └─ Если на дефолтной ветке — сначала ветка (см. git-workflow). Push только если founder попросил.
```

**Правило одного шага:** один проход ultracode = один шаг = один коммит. Не батчить шаги «чтобы быстрее».
Батчинг ломает атрибуцию багов и делает откат невозможным. Один зелёный коммит за раз — вот как достигается ZERO bugs.

## 3. Протокол чекпоинта — остановка и ожидание founder'а

Чекпоинты (🛑 ЧЕКПОИНТ A/B/C…) расставлены в файлах фаз в точках, где **раннее решение дорого менять позже**
(структура монорепо, порог покрытия, контракт golden-фикстуры, схема БД, дизайн-направление и т.п.).
На чекпоинте исполнитель **обязан остановиться** — даже если технически мог бы продолжить.

### Что делает исполнитель, дойдя до шага с 🛑

```
1. ВЫПОЛНИТЬ тело шага по полному циклу §2 (RED→GREEN→REFACTOR→GATES→COMMIT).
   Чекпоинт не отменяет TDD — сам шаг должен быть зелёным и закоммиченным.

2. ОСТАНОВИТЬСЯ. НЕ брать следующий шаг.

3. ЗАПИСАТЬ В STATE.md маркер остановки:
   - В «Текущая фаза»: строку `🛑 ОСТАНОВ НА ЧЕКПОИНТЕ <X> (после Шаг <n.m>) — ЖДУ founder'а`.
   - В «Журнал шагов»: обычная строка шага + пометка `[CHECKPOINT <X>]`.
   - В «Следующий шаг»: что именно пойдёт дальше, ЕСЛИ founder одобрит без правок.

4. ПОКАЗАТЬ founder'у краткую сводку (в ответе, не в файле):
   ┌─────────────────────────────────────────────────────────────┐
   │ 🛑 ЧЕКПОИНТ <X> — ОСТАНОВЛЕН, ЖДУ РЕШЕНИЯ                     │
   │                                                              │
   │ СДЕЛАНО (шаги n.a … n.m):                                    │
   │   • <буллеты: что построено, какие тесты зелёные>            │
   │ ДОКАЗАТЕЛЬСТВО ЗЕЛЁНОГО:                                     │
   │   • <вывод pnpm test / coverage exit-code / e2e>            │
   │ КЛЮЧЕВЫЕ РЕШЕНИЯ, КОТОРЫЕ ДОРОГО МЕНЯТЬ ПОЗЖЕ:               │
   │   • <напр. имена пакетов, порог 80%, форма golden-JSON>     │
   │ ЧТО ДАЛЬШЕ, ЕСЛИ ОДОБРИШЬ:                                  │
   │   • <следующий незакрытый шаг + его первый тест>            │
   │ ВОПРОСЫ К ТЕБЕ (если есть):                                 │
   │   • <развилки, требующие решения founder'а>                 │
   └─────────────────────────────────────────────────────────────┘

5. ЖДАТЬ. Не продолжать, пока founder явно не скажет «продолжай» / «go» / не внесёт правки.
```

### Что делает founder на чекпоинте

- **Одобряет** («продолжай») → исполнитель снимает маркер 🛑 в `STATE.md` и берёт следующий шаг по §2.
- **Правит** файл фазы / `docs/*` (меняет имена, пороги, контракт) → см. §6 (правило вмешательства).
- **Откатывает** → просит исполнителя вернуться к шагу N (git revert/checkout до коммита шага), переписать.

**Жёсткое правило:** исполнитель НИКОГДА не проходит 🛑 самостоятельно. Чекпоинт — это точка, где founder
дешевле всего меняет курс. Проскочить его = накопить дорогой долг.

## 4. ШАБЛОН ПРОМПТА для ultracode (запуск следующего шага/фазы)

> Founder копирует блок ниже в ultracode. Он самодостаточен: исполнитель сам находит, где «СЕЙЧАС», делает
> ровно следующий незакрытый шаг по TDD, обновляет `STATE.md` и останавливается на чекпоинте.
> Менять нужно только одну строку — `<ФАЗА>` опционально (можно оставить «авто»).

```text
Ты — исполнитель сборки FlipHouse. Работай строго по docs/05-ПРОТОКОЛ-ВЫПОЛНЕНИЯ-И-ТЕСТЫ.md.
Рабочая директория: /Users/mishanikhinkirtill/Desktop/FlipHouse. Все команды — из корня.

ЦЕЛЬ ФОУНДЕРА №1: ZERO bugs. Шаг не закрыт, пока тесты не зелёные И coverage-гейт держится.

ВЫПОЛНИ РОВНО ОДИН СЛЕДУЮЩИЙ НЕЗАКРЫТЫЙ ШАГ. Алгоритм:

1. ПРОЧИТАЙ STATE.md (корень). Найди раздел «Текущая фаза» и «Следующий шаг» — это «СЕЙЧАС».
   - Если в «Текущая фаза» стоит маркер `🛑 ОСТАНОВ НА ЧЕКПОИНТЕ` — НЕ продолжай:
     выведи краткую сводку чекпоинта (§3) и СТОП. Founder должен сказать «продолжай».
   - Иначе определи файл фазы roadmap/<ФАЗА>.md (или авто: фаза из STATE.md) и номер следующего незакрытого шага.

2. ПРОЧИТАЙ блок этого шага в файле фазы целиком: Цель/DoD, «Тесты СНАЧАЛА», «Реализация»,
   «✅ Готово когда», «Commit». Также сверься с релевантными docs/00..04, на которые ссылается шаг.

3. ВЫПОЛНИ ЦИКЛ §2 строго по пунктам:
   [RED] напиши падающий тест с ТОЧНЫМ именем/assertion из «Тесты СНАЧАЛА»; запусти точечно; убедись, что падает по делу.
   [GREEN] минимальная реализация; точечный прогон → зелёный; прогон слоя → соседи не сломаны.
   [REFACTOR] иммутабельность, мелкие файлы, DRY/KISS; после каждой правки — тесты слоя зелёные.
   [GATES] pnpm lint && pnpm typecheck && pnpm test && pnpm coverage (+ pnpm test:e2e если шаг трогает UI/flow).
           Любой красный гейт = шаг НЕ закрыт. Чини реализацию, не тест.
   [DoD] пройди список «✅ Готово когда» буквально, включая ручные проверки (напр. proof-of-failure exit-code).

4. ОБНОВИ STATE.md: добавь строку в «Журнал шагов» (phase.step | type | дата | что сделано | ✅ тесты зелёные),
   обнови «Текущая фаза» и «Следующий шаг» (= следующий незакрытый шаг). Новые риски → «Открытые риски».

5. СДЕЛАЙ ОДИН атомарный коммит с сообщением РОВНО из поля «Commit» шага (test+impl+STATE.md в одном коммите).
   Если на дефолтной ветке — сначала заведи ветку. Push только если я попросил.

6. ЕСЛИ следующий по порядку — шаг с 🛑 ЧЕКПОИНТ: выполни его тело по §2, затем запиши маркер
   `🛑 ОСТАНОВ НА ЧЕКПОИНТЕ` в STATE.md и ОСТАНОВИСЬ со сводкой §3. НЕ проходи чекпоинт сам.

ПРАВИЛА:
- Один проход = один шаг = один коммит. Не батчи шаги.
- TDD обязателен: тест ПЕРВЫМ, всегда видеть RED перед GREEN.
- Не трогай golden-файлы без явного `test(...): update golden` и обоснования.
- Если в файле фазы или docs/* появились правки от меня — приоритет у файла (см. §6): подхвати новый контракт,
  при конфликте с уже сделанным — отметь в STATE.md «Открытые риски» и спроси меня, прежде чем переделывать.

ВЫВЕДИ В КОНЦЕ: какой шаг закрыл, доказательство зелёного (вывод test/coverage), какой шаг следующий,
и стоишь ли ты на чекпоинте.
```

> Вариант «доделай фазу до следующего чекпоинта»: добавить в конец первой строки —
> «Повторяй шаги подряд по §2, пока не упрёшься в ближайший 🛑 ЧЕКПОИНТ, затем остановись (§3).»
> Это единственный санкционированный способ батчить — и он всё равно останавливается на чекпоинте.

## 5. Гейт релиза (перед деплоем на прод)

Деплой на прод (Railway) разрешён **только** после прохождения всего гейта ниже. Каждый пункт обязателен.

```
[ ] G1. ПОЛНЫЙ ПРОГОН ТЕСТОВ (зелёный, локально И в CI)
        └─ pnpm test:all  → lint + typecheck + все Vitest-проекты + pytest + coverage(пороги) + Playwright
```

## 6. Правило вмешательства — как founder правит фазу между шагами

Файлы фаз (`roadmap/P0..P7`) и `docs/00..04` — **живой контракт**. Founder вправе править их между любыми шагами
(особенно на чекпоинтах). Исполнитель ОБЯЗАН подхватывать правки. Механика:

### Как founder вносит правку

```
1. Между шагами (или стоя на 🛑) — отредактировать соответствующий файл фазы / docs:
   - изменить «Тесты СНАЧАЛА» (имя/assertion теста), «Реализацию», «✅ Готово когда», «Commit»,
     добавить/удалить/переставить шаг, сдвинуть чекпоинт, поменять порог/контракт.
2. Записать СУТЬ правки в STATE.md → «Открытые риски» или «Следующий шаг»
   строкой `[FOUNDER EDIT <дата>] <что и почему изменено>`, чтобы исполнитель её гарантированно увидел.
3. (опц.) Закоммитить правку файла фазы отдельным `docs:`-коммитом — так видно, что контракт сменился.
```

### Как исполнитель подхватывает (на каждом проходе §2 / §4)

```
1. ПЕРЕД взятием шага — ВСЕГДА перечитывать актуальный блок шага из файла фазы (не из памяти/кэша).
   Источник истины = ТЕКУЩЕЕ содержимое файла фазы, а не то, что было в прошлом проходе.
2. Сверить STATE.md на наличие `[FOUNDER EDIT ...]` со времени последнего своего обновления.
3. РАЗВИЛКА:
   • Правка касается ЕЩЁ НЕ сделанного шага → просто выполнить новый контракт по §2. Ничего переделывать не нужно.
   • Правка касается УЖЕ сделанного шага (изменился контракт/порог/тест задним числом)
     → НЕ молча переписывать. Записать в STATE.md «Открытые риски»:
       `[CONFLICT] правка шага <n.m> расходится с закоммиченным <commit>` и СПРОСИТЬ founder'а:
       переделать шаг (revert+redo по §2) или принять расхождение. Ждать ответа.
4. После подхвата — обновить STATE.md, что новый контракт принят (`[EDIT APPLIED <дата>] шаг <n.m>`).
```

**Принцип:** правки founder'а имеют приоритет над планом «в голове» исполнителя, но НИКОГДА не применяются
задним числом молча к уже зелёным коммитам — только через явный конфликт-флаг и решение founder'а.
Так курс меняется свободно, а ZERO-bugs-инвариант (каждый коммит зелёный и доказуем) не нарушается.

## Приложение: инварианты, которые держат ZERO bugs

1. **Каждый коммит зелёный.** Нет коммита без прохождения §2.GATES.
2. **Тест всегда первым.** RED обязан быть увиден до GREEN — иначе тест не доказывает поведение.
3. **Один шаг — один коммит.** Атомарность = откатываемость = локализуемость бага.
4. **Coverage-гейт роняет, а не варнит.** Доказано proof-of-failure тестом.
5. **Golden-файлы — священны.** Дрейф = RED; обновление = осознанный коммит с обоснованием.
6. **Чекпоинт = стоп.** Исполнитель никогда не проходит 🛑 сам.
7. **STATE.md — единственный источник истины о прогрессе.** Обновляется в каждом шаге, читается в начале каждого прохода.
8. **Файл фазы — источник истины о работе.** Перечитывается каждый проход; правки founder'а подхватываются по §6.
