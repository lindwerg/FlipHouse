# 03 — Офферы и вставка рекламы

> Это сердце FlipHouse и наш единственный реальный отрыв от рынка. Opus/Klap режут клипы, но не монетизируют. Rembrand вставляет рекламу, но не режет клипы и продаётся только брендам через «Request Demo». Whop/Vyro раздают клипы, но «реклама» — это ссылка в подписи. **FlipHouse — единственный, где нарезка клипа, подбор оффера и нативная вставка баннера в кадр — один self-serve цикл.** «Клип, который сам себя окупает».
>
> Этот документ описывает контракт между рекламодателем и движком: как бренд сдаёт ТЗ (схема оффера + UI-форма), как rules-engine превращает `оффер + клип → план размещения → FFmpeg-граф`, как работает brand-safety гейт, и как мы честно считаем показы и CPM.

```
                                       ▼  (клип «не eligible» для оффера)   
                                                                          
                                OFFER-RULES ENGINE                         
                                оффер + clip_meta → PlacementPlan          
                                (детерминированно, чисто, кешируемо)       
                                       │                                   
                                       ▼                                   
                                FFmpeg overlay-граф                        
                                (баннер вкомпонован в 9:16)                 
                                       │                                   
                                       ▼                                   
                                       └────────────────────►  4. публикует
                                                                  TikTok/Reels/Shorts
                                                                          
5.                          IMPRESSION / CPM АТРИБУЦИЯ                      
   платит по CPM/views      creator-OAuth метеринг + аудит        получает выплату
                            биллинг по дельтам просмотров          (фикс + CPM)
```

Ключевой принцип архитектуры: **rules-engine — чистая детерминированная функция** `plan(offer, clip_meta)`. Никакого I/O, никакого рандома, никаких обращений к часам. Одинаковый вход → байт-в-байт одинаковый filtergraph. Это делает движок тестируемым и кешируемым.

## 1. ТЗ рекламодателя — схема оффера (JSON Schema Draft 2020-12)

ТЗ — это машиночитаемый контракт, который компоновщик читает, чтобы решить: **какой** баннер, **куда**, **когда**, **поверх какого контента** и **за сколько**. Цвета — в `oklch`, чтобы брендовые цвета без потерь конвертации попадали в компоновщик через ту же токен-систему, что и сам продукт.

## 2. UI-форма рекламодателя («Для брендов»)

Пятишаговый визард, зеркалящий пять верхнеуровневых блоков схемы. Каждое поле мапится на JSON-путь, чтобы форма сериализовалась прямо в схему.

**Шаг 1 — Бренд** → `brand`
- Название бренда * → `brand.name` · Юр. лицо → `brand.legalName` · Сайт (https) → `brand.websiteUrl`
- Загрузка лого * (PNG RGBA / SVG, drag-drop) → `brand.logo.url` + `format` · Clear-space px → `minClearSpacePx`
- Основной цвет * (color picker → oklch) → `primaryColor` · Вторичный → `secondaryColor` · Текст-на-бренде (с AA-подсказкой контраста) → `contrastColor`
- Шрифт (+ тогл «дефолт платформы») → `fontFamily` · Вертикаль (select) → `verticalNiche`

**Шаг 2 — Ассеты баннеров** → `assets`
- Повторяемый загрузчик (1–12 строк): файл (image/*, .webm, .json/lottie), тип авто-детект → `type`/`url`; ширина/высота/длительность авто-читаются; тег аспекта (9:16, 1:1, …); авто-флаг прозрачности → `hasAlpha`; тогл loop. Валидатор: «Добавьте хотя бы один 9:16 или 1:1 прозрачный рендер».
- Аудио-стингер (≤3с, mp3) → `assets.audioStingerUrl`

**Шаг 3 — Размещение и тайминг** → `placement` + `timing`
- Позиция (визуальный 9:16-пикер с 9 слотами + «Auto») → `placement.position`
- Слайдер отступа от края 0–25% → `anchorMarginPct` · Слайдер max-покрытия 1–40% (живой превью размера на сэмпл-кадре) → `maxCoveragePct`
- Тоглы: «Никогда не перекрывать лица» (вкл) → `neverOverFace`; «Не перекрывать экранный текст» (вкл) → `neverOverText` · Face clearance px → `faceClearancePx`
- Редактор safe-zone / forbidden-band: перетаскиваемые прямоугольники на 9:16-превью; пресеты TikTok-рейл + нижний UI уже нарисованы → `safeZones` / `forbiddenBands`
- Поведение при коллизии (select) → `collisionPolicy`
- Старт/конец кампании → `timing.campaignStart`/`campaignEnd`
- Частота (radio: once / persistent / каждые N сек / on key moment) → `frequency`; условное поле «каждые N сек» для interval → `intervalSec`
- Задержка первого появления → `appearAtSec` · Длительность показа → `durationSec` · Max появлений на клип → `maxAppearancesPerClip` · Fade in/out ms → `fade`

**Шаг 4 — Сообщение и CTA** → `cta`
- Заголовок * (≤40, счётчик символов + превью читаемости в 9:16) → `headline` · Подтекст (≤60) → `subtext` · Кнопка (≤24) → `buttonLabel`
- Промокод (≤24) → `promoCode` · Destination URL (https) → `destinationUrl` · Метка рекламы (дефолт «Реклама»/«#ad») → `requiredDisclosure` · Языки копии (мульти-select) → `languages`

**Шаг 5 — Brand-safety, выплата, таргетинг** → `brandSafety` + `payout` + `targeting`
- Запрещённые категории контента (чипы из 18-значного enum) → `forbiddenCategories` · Стоп-слова конкурентов (tag input) → `blockedCompetitorTerms`
- Слайдер max-токсичности 0–1 → `minToxicityClearance` · Допустимые рейтинги (G/PG/PG-13/R) → `allowedRatings` · Тогл «ручная модерация каждого клипа» → `manualReviewRequired`
- Модель выплаты (radio: CPM / за 1k просмотров / фикс за клип / hybrid) → `payout.model`; условные поля: ставка за 1k → `rate`; фикс → `flatAmount`
- Валюта → `currency` · Бюджет → `totalBudget` · Min просмотров для зачёта → `minViewsToQualify` · Окно подсчёта (дни) → `viewWindowDays` · Комиссия платформы % (read-only, для прозрачности) → `platformFeePct`
- Ниши контента * (чипы) → `contentNiches` · Исключённые ниши → `excludedNiches` · Платформы (мульти-тогл) → `platforms` · Гео (мульти-select ISO 3166-1) → `geo` · Диапазон длительности клипа (dual slider) → `minClipDurationSec`/`maxClipDurationSec` · Уровни клипперов → `creatorTier`

**Submit** ставит `status: "in_review"`; brand-safety проверка оффера переводит в `active`. Поля `offerId`, `createdAt`, `updatedAt`, `platformFeePct` контролируются сервером и не редактируются рекламодателем.

## 3. Offer-rules engine: `оффер + клип → план размещения → FFmpeg-граф`

Чистая детерминированная функция. Порядок: **sanitize → выбор временных окон → пространственное решение → план → кодоген**.

### 3.1 Входные контракты

**Метаданные клипа (форма AutoFlip).** Все боксы — в координатах выходного канваса (после реврейма 1080×1920), поэтому движок никогда не рассуждает о crop-трансформе источника — это решает реврейм-стадия выше. Намеренная граница (KISS): движок видит только финальную 1080×1920 сцену.

```jsonc
{
  "clip_id": "clip_0007",
  "duration_s": 47.3, "fps": 30, "out_w": 1080, "out_h": 1920,
  "frames": [                                       // сэмплы, не каждый кадр (напр. каждые 0.25с)
    { "t": 0.0,
      "crop":         { "x":0,"y":0,"w":1080,"h":1920 },     // crop-бокс реврейма в OUT-координатах
      "caption_safe": { "x":60,"y":1380,"w":960,"h":420 },   // зона субтитров
      "speaker_box":  { "x":300,"y":520,"w":480,"h":620 }    // активный спикер (лицо)
    }
    // ...
  ]
}
```

### 3.2 Санитизация входа (fail fast на границе)

```
function sanitize(offer, clip):
    assert clip.out_w > 0 and clip.out_h > 0 and clip.duration_s > 0
    assert clip.fps in [1, 240]
    assert len(clip.frames) >= 2 and frames строго возрастают по t
    assert clip.frames[0].t == 0 and clip.frames[-1].t <= clip.duration_s + 1/fps

    for f in clip.frames:
        for box in [f.crop, f.caption_safe?, f.speaker_box?]:  // опциональные могут быть null
            clampBoxToCanvas(box, clip.out_w, clip.out_h)       // пересечение, никогда не доверяем сырому
            assert box.w >= 0 and box.h >= 0

    for b in offer.banners:
        assert fileExists(b.asset_path) and isRGBA(b.asset_path)
        assert 0 < b.w <= clip.out_w and 0 < b.h <= clip.out_h
        assert 0 < b.min_on_screen_s <= b.max_on_screen_s
        b.priority         = clampInt(b.priority, 0, 1000)
        b.animation        = whitelist(b.animation, {slide_up,slide_down,fade,pop}, default=fade)
        b.preferred_anchor = whitelist(b.preferred_anchor, {top,bottom,left,right,free}, default=bottom)

    offer.frequency.min_gap_s      = clampFloat(min_gap_s, 0, clip.duration_s)
    offer.frequency.max_concurrent = clampInt(max_concurrent, 1, 3)
    // asset_path НИКОГДА не интерполируется в shell-строку — передаётся как argv в ffmpeg -i
```

Жёсткие правила: отклоняем неизвестные enum-значения, а не пробрасываем их в FFmpeg; клампим каждый бокс в канвас (кривой `speaker_box` от плохого детектора не должен молча отключить проверку коллизий); пути к ассетам идут отдельными `-i` argv-записями — **никогда** не склеиваются в строку filtergraph (инъекция в filtergraph — это здешний аналог XSS).

### 3.3 Выбор временных окон (частота / интервалы)

```
function chooseWindows(offer, clip):
    windows = []
    cursor  = 0
    perMinuteBudget = rateLimiter(max_per_minute)
    for each banner b in offer.banners sorted by priority desc:
        placed = false
        for t0 in scanStarts(clip, offer.schedule_hint, step=0.5):
            if t0 < cursor + offer.frequency.min_gap_s:  continue   // интервал
            if not perMinuteBudget.canEmit(t0):           continue   // частотный кап
            dur = b.max_on_screen_s
            // ужимаем окно, пока не появится бесколлизионный слот, но не ниже min
            while dur >= b.min_on_screen_s:
                slot = solveSpatial(b, clip, t0, t0+dur)             // §3.4
                if slot.feasible:
                    windows.append(makeAssignment(b, t0, t0+dur, slot))
                    cursor = t0 + dur
                    perMinuteBudget.emit(t0)
                    placed = true; break
                dur -= 0.5
            if placed: break
    return enforceConcurrency(windows, offer.frequency.max_concurrent)
```

`scanStarts` кодирует `schedule_hint`: `even` → равномерная сетка; `front_load` → смещение к малым t; `on_silence` → старты только в кадрах, где `speaker_box` отсутствует (никто не говорит).

### 3.4 Пространственное решение — избегание коллизий (ядро)

Для окна `[t0,t1]` берём **объединение всех боксов-препятствий по кадрам окна** (worst-case occupancy — баннер обязан быть безопасен в *каждом* кадре своей видимости, поэтому берём union, а не per-frame). Затем ищем якорь, чей прямоугольник не задевает препятствия.

```
function solveSpatial(b, clip, t0, t1):
    obstacles = emptyBoxList()
    for f in framesInRange(clip, t0, t1):
        if not b.allow_overlap_caption and f.caption_safe: obstacles += f.caption_safe
        if not b.allow_overlap_face    and f.speaker_box:  obstacles += f.speaker_box
    obUnion = mergeBoxes(obstacles)

    M = 24                                  // отступ в px (именованная константа, реф 1080w)
    candidates = anchorRects(b, clip, M, order = preferenceOrder(b.preferred_anchor))
    // напр.: bottom-center над субтитрами, затем top-center, затем грубая сетка-фолбэк
    for rect in candidates:
        if insideCanvas(rect, clip) and noOverlap(rect, obUnion, pad=M):
            return { feasible:true, rect:rect, anchor:rect.anchor }
    return { feasible:false }

noOverlap(a, b, pad) =
   a.x+a.w+pad <= b.x  OR  b.x+b.w+pad <= a.x  OR
   a.y+a.h+pad <= b.y  OR  b.y+b.h+pad <= a.y       // AABB-разделение с отступом
```

`caption_safe` и `speaker_box` — **жёсткие** ограничения, если баннер явно не разрешил `allow_overlap_*`. Если ни один якорь не влезает на полной длительности — §3.3 уже ужимает длительность; если и тогда ничего — баннер **дропается для этого клипа** (с логом причины), а не накладывается на лицо. Детерминированно и консервативно.

### 3.5 Анимация → функция смещения

Каждая анимация — закрытая форма смещения `(dx,dy,alpha)(τ)`, где `τ` — локальное время от `t0`, с IN/OUT-рампами `R=0.35с`. Покоящаяся позиция — `rect.(x,y)`.

```
slide_up:   старт 80px ниже покоя, easing к покою за R; реверс на OUT
fade:       dx=dy=0; alpha 0→1 за R, 1→0 на OUT
pop:        scale 0.6→1 (мапится в dx,dy от центра) за R через overshoot-lerp
```

Эти выражения мапятся в FFmpeg-expr (`lerp`/`clip`) и в MoviePy-колбэки одинаково — те же числа, два бэкенда.

### 3.7 Кодоген в FFmpeg overlay-граф

> Все FFmpeg-паттерны ниже **эмпирически выполнены и подтверждены на FFmpeg 8.1**. Ловушка прода: `drawtext` требует `--enable-libfreetype`, которого нет в стоковом Homebrew-билде. **Поэтому ядро движка — баннер как заранее отрендеренный PNG/RGBA + `overlay`**: быстрее и убирает любой риск экранирования текста.

Три несущих примитива:
- **`overlay` + `enable='between(t,a,b)'`** — гейт по времени. Вне окна кадры проходят нетронутыми.
- **`eval=frame`** — обязателен, если `x`/`y` зависят от `t`. `eval=init` замораживает позицию на t=0.
- **Анимированная alpha** — у `overlay` нет своей; затухание делаем на *входе баннера*: `format=rgba,fade=...:alpha=1` с `-loop 1`, чтобы у одного PNG-кадра была реальная таймлиния.

Кодоген обходит `placements[]` и эмитит **один `overlay`-узел на баннер**, цепочкой. Реврейм/субтитры дают `[base]`; баннеры цепляются после.

**Входы (argv — никогда не интерполируем строкой):**
```
ffmpeg -i reframed_clip_0007.mp4 \
       -loop 1 -i assets/cta_signup.png \
       -filter_complex_script plan_0007.filtergraph \
       -map "[vout]" -map 0:a -c:a copy out_0007.mp4
```

**Сгенерированный `plan_0007.filtergraph`** (для примера; `R=0.35`, покой `x=330`, `y=1236`, slide от `dy=80`):
```
[1:v]format=rgba,
     fade=t=in:st=6.0:d=0.35:alpha=1,
     fade=t=out:st=10.65:d=0.35:alpha=1 [bnr0];

[0:v][bnr0]
  overlay=
    x=330:
    y='1236 + 80*(1 - clip((t-6.0)/0.35,0,1))
            + 80*clip((t-10.65)/0.35,0,1)':
    eval=frame:
    enable='between(t,6.0,11.0)'
  [vout]
```

Как используется каждый подтверждённый примитив:
- `enable='between(t,6.0,11.0)'` → баннер компонуется только внутри своего окна.
- `y`-выражение → `clip((t-t0)/R,0,1)` рампит 0→1 на in-рампе (slide-up); второй член отыгрывает смещение на out-рампе. Только подтверждённые `clip`/`+`/`*`.
- `fade ...:alpha=1` на входе баннера обрабатывает прозрачность независимо от позиции (чище, чем паковать alpha в overlay).
- `eval=frame` — `x`/`y` пересчитываются покадрово (анимация реально движется).
- Несколько баннеров → дописываем `[vout_k][bnr_{k+1}] overlay=...[vout_{k+1}]`, финальный пад `[vout]`. `max_concurrent` из §3.3 гарантирует, что одновременно перекрывается не больше N.

**Безопасность filtergraph (эмпирически доказано):** наивная интерполяция недоверенных данных оффера в строку фильтра позволяет `,` `;` `:` `=` `[` `]` `'` `\` `%` вырваться из параметра и инжектнуть новые фильтры (`overlay=x=0,split:y=0` ломает граф). Митигация: (1) **клампим/whitelist'им каждый числовой вход** до интерполяции; (2) недоверенный текст графа **никогда** не идёт на shell/`-filter_complex` — только `-filter_complex_script FILE`; (3) для текста — `drawtext=textfile=FILE` (грузит байты литерально) либо, лучше, рендер текста в PNG вне графа; `fontfile` — только из allowlist, callerы его не задают.

**MoviePy v2 — фолбэк** (те же числа, для окружений без libfreetype-FFmpeg или для сложной per-frame Python-логики; медленнее — полный decode/re-encode в Python):
```python
def pos(t, p=placement):                      # подтверждено: with_position принимает t->(x,y)
    tau  = t - p.t_start
    rise = 1 - clip(tau / p.ramp_s, 0, 1)
    return (p.rect.x, p.rect.y + p.from_offset.dy * rise)

banner = (ImageClip(p.asset_path)             # RGBA-прозрачность уважается автоматически
          .with_start(p.t_start)
          .with_duration(p.t_end - p.t_start)
          .with_position(pos)
          .with_effects([vfx.CrossFadeIn(p.ramp_s), vfx.CrossFadeOut(p.ramp_s)]))
final = CompositeVideoClip([base, *banners])
```
В MoviePy инъекция схлопывается до обычной валидации Python-значений (всё равно клампим координаты/длительности, всё равно allowlist путей ассета/шрифта).

## 4. Brand-safety гейт

Гейт прогоняется **один раз на клип, до того как клип становится eligible для оффера**. **Fail-closed:** если любой анализатор падает или скор за порогом — клип уходит на ручную модерацию, а не авто-листится.

```
INGEST CLIP
   │
   ├─ 1. Сэмплинг кадров (ffmpeg: 1 fps + keyframes сцен-катов)
   │
   ├─ 2. ВИЗУАЛЬНАЯ БЕЗОПАСНОСТЬ (на сэмпл-кадр)
   │     ├─ NSFW/nudity классификатор  → reject если porn/sexy/hentai > τ
   │     └─ Violence/gore классификатор → reject если violent > τ
   │
   ├─ 3. FACE / REGION MAP (на кадр)   ← безопасность РАЗМЕЩЕНИЯ баннера, не reject
   │     └─ Боксы лиц + персон → per-frame «no-banner zones» (это и есть speaker_box для §3.4)
   │
   ├─ 4. AUDIO → ТРАНСКРИПТ
   │     ├─ VAD → ASR (word-level timestamps)
   │     └─ Profanity/toxicity по транскрипту → flag/reject по severity
   │
   └─ 5. РЕШЕНИЕ
         APPROVE  → клип eligible, сохраняем safe-боксы размещения
         REVIEW   → у порога или ошибка анализатора (fail-closed)
         REJECT   → жёсткое нарушение
```

Критично разделять две вещи: стадии 2 и 4 — **гейты eligibility** (может ли баннер тут вообще быть). Стадия 3 — **ограничение размещения** (где на каждом кадре баннер может сидеть); она производит боксы лиц/персон, которые overlay-движок (§3.4) вычитает, чтобы держать баннеры подальше от лиц. Не путать.

**Гейт — это гейт, а не фильтр:** если классификатор клипа задел любую `forbiddenCategories`, превысил `minToxicityClearance` или совпал с `blockedCompetitorTerms` — оффер просто не отдаётся под этот клип. Клиппер видит это как «not eligible».

## 5. Impression / CPM атрибуция — честно

### 5.1 Главная правда архитектуры

Как только клип с вкомпонованным баннером опубликован в TikTok/YouTube/Reels — **у нас нулевая прямая телеметрия**. Баннер вшит в пиксели, это не трекаемый ad-unit. Платформа считает просмотр у себя и не говорит нам ничего. Значит, «показы нашего баннера» можно только **вывести из просмотров хост-видео**, который мы получаем одним из каналов ниже.

**Самый важный факт:** на всех трёх платформах надёжные per-video просмотры требуют, чтобы **креатор один раз авторизовал наше приложение через OAuth**. Нет коммерческого масштабируемого способа прочитать просмотры чужого видео, кроме публичного lifetime `viewCount` у YouTube. Это формирует всю v1 — и это не ограничение, а **enabler**: OAuth-грант превращает «нетрекаемые off-platform пиксели» в «метрируемую биллабельную поверхность».

| Канал | Per-video просмотры? | Реальность гейтинга | Вердикт v1 |
|---|---|---|---|
| **TikTok Display API** (`/v2/video/list/`, scopes `video.list` + `user.info.basic`) | Да — `view_count` и др. | Креатор OAuth'ит через Login Kit; возвращает только видео авторизовавшего | **Основной сигнал (TikTok)** |
| TikTok Research API | Да | «Creator/advertiser/commercial → not eligible» | **Коммерчески недоступен** |
| **YouTube Analytics API** (`reports.query`, `views`/`video`) | Да, per-video time-series | OAuth владельца канала; **не нужен** Partner/Content Owner | **Основной сигнал (YouTube)** |
| YouTube Data API `videos.list.statistics.viewCount` | Да — публично, без auth | Только lifetime cumulative, без time-series | **Фолбэк / аудит-кросс-чек** |
| **Instagram Graph API** (Reels `views`/`plays`, `reach`) | Да | Business/Creator + связанная FB Page + OAuth; insights с 1k+ подписчиков | **Основной сигнал (Reels)** |
| Caption tracking links / UTMs | Нет (меряет *клики*, не просмотры) | TikTok/IG подписи ненадёжно кликабельны | **Только click-атрибуция (другой продукт)** |
| Уникальный watermark + OCR | Нет (идентичность, не счёт) | — | **Фрод/диспут-инструмент, не счётчик** |

### 5.2 Модель v1 — Creator-OAuth метеринг с публичным аудитом

```
1. CREATOR ONBOARD
   Креатор коннектит TikTok (Login Kit) / YouTube (Google OAuth) / IG (Meta OAuth).
   Refresh-токены шифруем (KMS). Scopes: минимальные read-only stats.

2. INSERT + REGISTER
   При вставке баннера в клип C для рекламодателя A:
   - генерируем impression_unit_id (campaign × placement × clip)
   - вшиваем низковидимый/стегано watermark-токен в рендер
   - персистим: render hash, рекламодатель, CPM-ставку, % экранного времени баннера, in/out timestamps

3. PUBLISH CLAIM
   Креатор публикует. Ловим URL/ID:
   - авто: опрос video.list (TikTok) / uploads (YT) / media (IG) + fingerprint-матч по render hash + watermark
   - ручной фолбэк: креатор вставляет URL (с аудитом, шаг 6)

4. METER (собственно счёт)
   Опрос per-video просмотров по затухающему расписанию:
     hourly день 1 → 4×/день до дня 7 → daily до дня 30 → weekly до дня 90 → freeze.
   Храним time-series снапшоты, биллим по ДЕЛЬТАМ.

5. КОНВЕРТАЦИЯ просмотры → биллабельные показы
   billable_impressions(window) = Δviews × banner_visibility_factor
     где visibility_factor = экранное время баннера ÷ длительность клипа (1.0 для persistent overlay)
   charge = billable_impressions / 1000 × CPM_rate
   Стоп счёта на freeze дня 90. Settle помесячно по дельтам.

6. AUDIT / FRAUD (параллельно, не блокирует биллинг)
   - кросс-чек OAuth view_count против публичного YouTube viewCount (единственный независимый источник)
   - velocity-аномалии (просмотры растут физически невозможно / плоско-затем-вертикально → флаг view-фрода)
   - watermark/OCR на диспутах (доказать, что баннер реально был в кадре)
   - discrepancy > порога (напр. 15%) → hold settlement, ручной разбор
```

### 5.3 На чём биллим (механика CPM)

- **Биллим по дельтам просмотров** в окне счёта, не по lifetime — чтобы перечитанный устаревший клип не списался дважды.
- **«Показ» = просмотр хост-видео в окне, когда баннер был на экране**, масштабированный `banner_visibility_factor`. Делаем это определение **контрактным** с рекламодателем заранее — это крупнейший источник будущих диспутов.
- **Freeze на дне 90.** Long-tail просмотры реальны, но несобираемы в чистом цикле и создают бесконечную открытую ответственность.
- **Settle помесячно по накопленным дельтам**, любой клип с флагом аудита — на hold.

### 5.4 Честные ограничения (не заметаем под ковёр)

1. **Это view-атрибуция, а не viewability-верификация.** Биллим за «хост-видео посмотрели», не за «глаза человека были на пикселях баннера». Нет MRC-grade viewability, нет IAB SIVT/GIVT-фильтрации. Это ближе к influencer/sponsorship CPM, чем к programmatic display. **Позиционируем как influencer-grade view-attributed CPM, а не verified-display CPM** — честный фрейминг, который снимает диспут, способный убить renewals.
2. **Нет покрытия для неонбордженных креаторов.** Репост/стич/дуэт/перезалив тем, кто не OAuth'ил нас — невидимы. Наши числа — это **пол**, так и описываем.
3. **Определения просмотра у платформ разные и плавают.** TikTok view ≠ YouTube view ≠ Reels play. Нормализуем по платформе и раскрываем методологию; никогда не подаём один смешанный «impressions» как apples-to-apples.
4. **View-фрод реален, и мы ниже по течению от него.** Купленные просмотры раздувают и API-числа. Аудит-слой смягчает, но не устраняет; YouTube-кросс-чек — единственный независимый второй источник, у TikTok/IG независимого коммерческого кросс-чека **нет**.
5. **OAuth-токены гниют.** Креаторы отзывают, аккаунты банятся, токены истекают. Строим graceful degradation к публичному `viewCount` (только YouTube) + ручной creator-reported с аудитом.
6. **Watermark/OCR — это идентичность, не метеринг.** Доказывает *что баннер был в видео*, но не *сколько людей его видели*. Не давать команде путать одно с другим.

**Отложено в v2+:** IAB/MRC-aligned viewability-прокси (completion-rate-weighted показы через `average_view_duration`), repost-discovery через Content-ID-style fingerprint по неонбордженным аккаунтам, SIVT/GIVT-фильтрация невалидного трафика, demo/geo-разбивки для таргетированных CPM-тиров.

## 6. Чем это нас делает лучшим на рынке

Три отдельных бизнеса трогают «клипы», и никто не владеет швом между ними:

| Слой | Кто | За что берёт | Чего НЕ делает |
|---|---|---|---|
| **A. AI-нарезка (SaaS)** | Opus.pro, Klap, Vizard, Submagic | $15–39/мес подписка, метрика по минутам *источника* | Не монетизируют, нет рекламодателей, нет вставки оффера. Чистый cost-центр для креатора. |
| **B. Клиппер-армии (агентства)** | Whop, Vyro, MrBeast/Adin Ross программы | $2–8 CPM клипперам; агентства $2.5k–10k/мес | «Реклама» = ссылка в подписи. Нет нативного in-video оффера, view-фрод. |
| **C. AI in-video вставка** | **Rembrand** (реальная угроза), Keek (patent-pending, не запущен) | CPM, креаторам 75% net | Rembrand продаётся **enterprise через demo**, не self-serve long-tail, и **не генерит клипы**. |

**Зазор, которым владеет FlipHouse:** никто не сливает A+B+C в один self-serve цикл. Мы — единственные, кто говорит: *залей длинное видео → AI режет клипы → AI авто-вставляет подобранный оффер в клип → рекламодатель платит → клиппер получает, без агентского посредника.*

Три структурных клина:
1. **Бандлим cost-центр с revenue-центром.** Тот же акт (нарезка) становится моментом монетизации. У Opus причина — «вдруг завирусится»; у FlipHouse — «мне платят за клип, гарантированный пол + CPM-апсайд».
2. **Self-serve двусторонний маркетплейс для long-tail, который Rembrand игнорит.** Self-serve онбординг выплат для креатора с 5k подписчиков и DTC-бренда с бюджетом $500. Matching-движок (оффер ↔ тема/аудитория клипа через transcript-эмбеддинги) — это и есть продукт.
3. **Нативная in-clip вставка оффера, не bio-ссылка.** Механика Rembrand, но автоматически в момент нарезки и выбранная матчем маркетплейса.

**Компаундящиеся моаты:** (1) data-flywheel качества матчей — единственные с fused clip+advertiser+performance данными; (2) verified-views settlement (§5) — решает проблему №1 доверия в клиппинге; (3) brand-safety/consent-гейтинг (§4) с авто-disclosure под FTC/ASA — поднимает барьер для fast-followers; (4) self-serve advertiser-консоль — то, что Rembrand прячет за «Request Demo».

**Риски честно:** Rembrand хорошо профинансирован ($68.5M, The Trade Desk + L'Oréal BOLD на капе) — бьём self-serve + long-tail + бандлом нарезки, не вставкой как таковой (вставку они догонят). Авто-вставленная реклама **обязана** нести FTC/ASA disclosure, иначе TikTok/YouTube/Meta снесут аккаунты — disclosure вшит в пайплайн как несъёмная фича (см. `cta.requiredDisclosure`). Cold-start двусторонности убивает стартапы — de-risk через concierge/agency-first фазу (мы *становимся* нарезчик-агентством, минус их 30% cut).

## Сводка ключевых правил, которые компоновщик читает из этого контракта

- **never-over-face** = `placement.neverOverFace` + `faceClearancePx` → детектор лиц вычитает эти регионы из кандидатных слотов до размещения (§3.4, `speaker_box`).
- **safe-zones / forbidden-bands** — нормализованные rect'ы, поэтому один оффер компонуется корректно через UI-оверлеи TikTok/Reels/Shorts без переавторинга.
- **brand-safety — это гейт, не фильтр:** задел `forbiddenCategories` / превысил `minToxicityClearance` / совпал с `blockedCompetitorTerms` → оффер не отдаётся под клип, клиппер видит «not eligible».
- **payout-условия** (`if/then` в `allOf`) гарантируют, что `rate` есть для view-моделей и `flatAmount` для flat/hybrid — кривой оффер не доедет до расчёта выплаты.
- **детерминизм движка** (§3): нет часов/RNG/сети; сортировка `(priority desc, t asc)` тотальна → стабильный, кешируемый, диффабельный выход; инфизибельные баннеры уходят в `dropped[]` с причиной, а не деградируют в наложение на лицо.
